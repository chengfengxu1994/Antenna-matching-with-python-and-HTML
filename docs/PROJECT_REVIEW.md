# 项目完整 Review（产品 + 总体工程）

Review 日期：2026-07-14

## 结论

项目已经不是概念验证：具备 SNP/S2P 数据接入、真实离散元件、多端口、联合优化、多场景、效率链路、Smith/S11 展示和手动调谐，也有一套数值质量较高的独立核心包。当前最大问题不是“功能少”，而是两套算法核心、四代 API 和巨型 UI/服务文件同时存在，产品能力超过了工程骨架的承载能力。

建议产品定位先收敛为“本地 RF 匹配分析工作台”，暂不把它当成多人 Web 平台。近期成功标准应是：结果可信、流程可复现、任务可恢复、版本可比较，而不是继续增加新的优化模式。

## 产品链路

现有主链路基本完整：选择数据目录 → 加载 DUT → 配置端口/频段/目标 → 自动优化 → 比较候选 → 扫频/功率平衡 → 手动微调。多 SNP 共用元件方案是明确的差异化能力。

仍缺少的产品闭环：

- 项目文件：当前设置和结果只在内存中，关闭程序后不可恢复。
- 可追溯报告：缺少输入文件哈希、算法/元件库版本、参数、候选列表和曲线的一键导出。
- 长任务体验：没有稳定的进度、取消、后台运行和失败恢复。
- 结果解释：需要统一说明 total efficiency、radiation efficiency、accepted power、coupling 和 component loss 的口径。
- 决策对比：候选方案应支持钉选后并排比较 BOM、最差频点、容差良率与各场景退化。
- 错误体验：前端仍使用 `alert` 或静默 `catch`，用户无法区分目录错误、格式错误、无候选和后端离线。

## 工程发现

### P0：正确性与交付

- Git 仓库目前没有任何提交，全部文件都处于 untracked 状态；这意味着没有可回退基线，也无法可靠审计改动。确认大文件和历史备份的去留后，应立即建立第一个基线提交。
- README 宣称的一键启动文件曾缺失，已补齐跨平台 `start.py` 与 Windows 入口。
- 效率接口的前端发送 JSON，但后端原先把字段声明为 query 参数；全局效率赋值又被函数局部变量遮蔽。接口可能返回异常或“成功但未生效”，现已统一为请求模型和集中状态。
- SNP 主加载接口原先未使用已有的目录边界校验，现已修复路径穿越风险。
- 数据库候选路径硬编码到 `E:\RF matching`，现改为相对当前工作区并支持环境变量。
- 服务原先默认监听所有网卡且 CORS 全开放；本地产品没有认证能力，现默认仅监听回环地址并限制开发源。

### P1：架构与维护性

- `api/server.py` 仍约 1800 行并保留多代 endpoint；建议按 router 拆分，但应在 API contract test 到位后进行。
- `TuningPanel.jsx` 超过 1100 行，混合表单、任务编排、结果表、图表和手调状态；建议拆为 `useTuningJob`、`TuningSetup`、`CandidateTable`、`ResultCharts`、`ManualEditor`。
- `apps/api/engine` 与 `packages/rfmatch-core` 重复实现 network、touchstone 和 optimizer。继续双线演进会出现同名概念不同口径。
- 全局单例状态符合本地单用户产品，但不支持多 worker、多用户或并发任务；部署文档必须明确这一点。
- 根级 `pytest.ini` 已明确同时收集产品测试与 `packages/rfmatch-core/tests`，人工脚本不进入门禁；当前根目录直接执行 `python -m pytest` 即覆盖两层测试，避免只跑产品层形成假绿色。

### P2：算法与性能

- 产品目标中的“快速、准确、无遗漏”需要拆开定义。Beam search 可以快，但不能证明无遗漏；完整离散 S2P 组合在高元件数下也不现实。
- 应以可验证指标替代绝对承诺：固定数据集 top-k recall、与 exhaustive 小规模结果差、运行时间、功率守恒误差、跨版本 golden 差异。
- 推荐搜索策略：可行拓扑语法 → 物理有效的下界剪枝 → 多起点连续优化 → 离散映射 → 邻域精修 → 小规模 exhaustive 校准。
- 所有性能优化必须复用同一个完整多端口 evaluator，禁止用与最终目标不一致的局部 S11 评分剪枝。

## 分阶段路线图

### 第一阶段：可信与可交付

- 固化统一 `/api/tuning/*` 契约并增加 API 测试。
- 建立 5–10 个真实 golden case，覆盖单端口、多端口、耦合、效率文件和多场景。
- 增加项目保存/加载和结果导出。
- 给长任务增加 job id、进度和取消。

### 第二阶段：收敛核心

- 用 adapter 将 Web 产品逐段迁移到 `packages/rfmatch-core`。
- 对新旧引擎做逐频点双跑，误差过线才切换。
- 删除 legacy endpoint 与重复计算模块。

### 第三阶段：工程化优化

- 引入有效下界的 branch-and-bound 或经过 recall 基准验证的混合搜索。
- 增加容差/良率排序、BOM 约束和方案对比报告。
- 若需要团队部署，再实现认证、会话持久化、任务队列和数据权限。

## 本轮落地

- 按 `apps / packages / data / tests / scripts / archive / artifacts` 重新归纳项目，历史与运行产物不再与现役代码并列。
- 将项目路径集中到 `apps/api/project_paths.py`，测试、日志、数据库和维护脚本共用同一路径口径。
- 新增可运行启动入口和便携路径配置。
- 请求模型与会话状态从巨型 server 中抽离。
- 修复效率 API 状态与 JSON 契约。
- 修复 SNP 安全路径与本地服务暴露面。
- 新增 API 基础回归测试和架构文档。

下一轮不建议直接大拆 `server.py` 或算法核心；应先增加 endpoint contract 测试和 golden 双跑，避免“结构变好但数值悄悄变化”。
